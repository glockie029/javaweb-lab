# JavaWeb 代码审计检查清单

该清单用于进入真实项目前，建立一套固定、可重复执行的排查顺序。推荐按顺序执行，避免只盯业务接口而忽略依赖、配置、容器和中间件。

## 1. 工程与依赖识别

1. 项目使用 `Maven` 还是 `Gradle`，启动模块在哪里。
2. 打包方式是 `jar`、`war` 还是多模块混合部署。
3. 是否存在父子模块、公共模块、网关模块、后台管理模块、测试模块。
4. 核心依赖是什么：`Spring Boot`、`Spring MVC`、`Spring Security`、`Shiro`、`MyBatis`、`JPA/Hibernate`、`Jackson`、`Fastjson`、模板引擎等。
5. 是否存在历史高风险组件或明显过旧版本。
6. 是否存在 profile、资源过滤、外部配置加载和测试配置残留。

## 2. 启动与容器入口

1. 项目启动方式是什么，主启动类或容器配置在哪里。
2. 是否部署在外部 `Tomcat`，还是使用嵌入式容器。
3. 入口点位于 `Servlet`、`Filter`、`Listener`、`Controller`、消息监听器，还是定时任务。
4. 是否存在全局异常处理、统一响应包装、统一鉴权和统一日志处理。
5. 是否暴露 `Actuator`、调试接口、测试接口、健康检查或管理接口。

## 3. 请求入口与参数来源

1. 参数来自 `query string`、表单、JSON、Header、Cookie、Path Variable、文件上传还是消息体。
2. 参数是否经过类型校验、长度校验、格式校验、白名单校验和业务校验。
3. 参数绑定是否依赖框架自动转换、反序列化、Bean 绑定、批量赋值或对象映射。
4. 是否存在用户输入直接进入路径、SQL、模板、脚本、URL、文件名、表达式的情况。

## 4. 认证、授权与会话

1. 登录校验在什么位置执行，是否存在白名单、忽略路径或顺序错误。
2. Session 是否在登录后重新生成，退出后是否彻底失效。
3. Token、Cookie、Session 是否存在固定值、弱校验或过期处理缺陷。
4. 接口是否只检查“是否登录”，未检查“是否有权访问该资源”。
5. 是否存在水平越权、垂直越权、菜单控制替代后端鉴权的情况。
6. 是否存在开发接口、批量操作、导出接口、管理接口未做保护。

## 5. 数据库与持久层

1. SQL 是否存在字符串拼接、动态表名、动态排序、动态条件未做限制。
2. 是否统一使用预编译语句，ORM 是否存在原生 SQL、HQL、JPQL 拼接。
3. 是否存在批量更新、批量删除、条件缺失、逻辑删除绕过等高风险场景。
4. 异常信息是否把 SQL 错误、数据库结构和绝对路径直接返回前端。
5. 数据权限是否在查询层、服务层或控制层存在缺失。

## 6. 文件、路径与对象处理

1. 文件上传是否校验扩展名、MIME、内容类型、大小、保存路径和重命名策略。
2. 文件名、目录名、压缩包内容是否可控，是否存在目录穿越和 `Zip Slip` 风险。
3. 下载、读取、删除、预览文件时是否直接拼接用户输入路径。
4. 是否存在模板文件、脚本文件、配置文件、类文件可写入风险。
5. 是否存在对象流读取、缓存对象恢复、自定义协议解析和不安全反序列化入口。

## 7. 输出、模板与前端交互

1. 页面输出是否对用户输入做 HTML、属性、URL、JavaScript 上下文编码。
2. 返回 JSON、HTML、富文本、Markdown、模板片段时是否区分安全输出场景。
3. 模板引擎、表达式引擎、EL、SpEL、OGNL 是否接收可控输入。
4. 跳转地址、模板变量、富文本渲染是否存在 XSS、SSTI 或开放重定向风险。

## 8. 外部请求与网络能力

1. 是否存在根据用户输入发起 HTTP、HTTPS、FTP、LDAP、JDBC 等外连请求的能力。
2. URL、Host、端口、协议、路径是否可控，是否存在 SSRF 风险。
3. XML 解析、SOAP、SAML 或第三方回调处理中是否关闭外部实体和危险特性。
4. 是否存在任意 webhook、回调转发、下载远程文件或代理访问功能。

## 9. 危险能力调用

1. 是否存在命令执行、脚本执行、运行时编译、动态类加载、反射调用。
2. 是否存在 `Runtime.exec`、`ProcessBuilder`、脚本引擎、类加载器、自定义表达式执行。
3. 是否存在动态 Bean 名称、动态方法名、动态路由、动态模板路径。
4. 是否存在 `JNDI/LDAP/RMI` 等高风险远程命名或对象获取能力。

## 10. 配置与中间件

1. `application.yml/properties`、`bootstrap`、`logback.xml`、`web.xml`、安全配置文件是否存在敏感信息或危险开关。
2. 是否存在默认口令、硬编码密钥、调试模式、开发环境配置残留。
3. 数据库、Redis、消息队列、对象存储、搜索引擎、邮件服务配置是否暴露。
4. 是否存在上传目录、临时目录、缓存目录映射到可访问路径。
5. 反向代理头、真实 IP 识别、转发头信任是否存在绕过风险。

## 11. 常见 Web 漏洞专项

1. 是否存在 SQL 注入、XSS、文件上传、路径穿越、命令执行、反序列化。
2. 是否存在 XXE、SSRF、CSRF、CORS 配置错误、开放重定向。
3. 是否存在表达式注入、模板注入、任意类加载、敏感信息泄露。
4. 是否存在未授权访问、越权访问和接口级数据泄露。

## 12. 日志、异常与敏感信息

1. 日志中是否记录口令、令牌、验证码、身份证号、银行卡号或内部路径。
2. 异常栈、SQL 语句、框架版本、类路径、服务器路径是否直接返回给用户。
3. 是否存在 `.env`、备份配置、历史配置、副本文件或密钥材料泄露。

## 13. 验证与利用条件

1. 风险是否需要登录、特定角色、特定业务状态或特定部署方式。
2. 风险是否依赖某个 profile、外部配置、容器版本、插件版本或组件开关。
3. 风险是否可稳定复现，是否存在破坏性影响，是否需要隔离验证环境。

## 14. 审计输出建议

每发现一个问题，至少记录以下内容：

1. 问题名称与漏洞类型。
2. 代码位置与关键调用链。
3. source、校验点、sink。
4. 触发条件、权限要求、部署前提。
5. 影响范围、风险等级和利用方式。
6. 修复建议与回归验证点。
