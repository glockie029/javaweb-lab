### 阶段五：数据交互、输入输出与持久层

状态：已完成

学习目标：

1. 看懂 JDBC、SQL 执行、参数到数据库的流转关系，以及数据库访问为什么是代码审计高频入口。
2. 看懂文件上传、文件下载、路径处理与输出点为什么同样属于阶段五核心内容。
3. 看懂 `MyBatis` 在 JavaWeb 项目中的位置、执行流程以及 `#{}` / `${}` 的风险差异。

涉及技术栈或组件：

1. JDBC
2. SQL
3. 数据库连接
4. JavaWeb 服务端数据访问链
5. 文件上传 / 文件下载
6. 路径处理
7. 输出点 / XSS
8. MyBatis
9. H2 内存数据库

本阶段已学习内容：

1. 已完成 JDBC 基础认知，包括 `Connection`、`Statement`、`PreparedStatement`、`ResultSet` 的职责和基本差异。
2. 已补充持久层风险总览，明确持久层风险不只包含 SQL 注入，还包括动态 SQL、数据越权、条件缺失、异常泄露和 ORM 风险。
3. 已结合样例理解“查询类数据越权为什么常发生在持久层”，并在当前项目中实现脆弱/安全查询对比。
4. 已完成文件上传、文件下载和路径穿越的演示代码，理解用户输入如何影响最终读取路径和写入路径。
5. 已完成输出点与 XSS 的最小演示代码，理解用户输入直接进入 HTML 输出时的风险。
6. 已完成 `MyBatis` 基础流程学习，理解 `Mapper 接口 -> Mapper XML -> SQL -> 对象映射` 的执行链。
7. 已结合当前项目实现 `MyBatis + H2` 最小示例，掌握 `#{}` 与 `${}` 的根本区别。
8. 用户对事务与连接池已有基础，本阶段未做展开式讲解，但可视为已具备基础认知。

关键调用链或结构理解：

1. 入口点：用户参数从 Servlet、表单、查询字符串或文件上传表单进入后端逻辑。
2. 中间处理组件：JDBC 连接、SQL 构造、参数绑定、文件路径拼接、模板/HTML 输出、MyBatis Mapper 代理与 XML SQL。
3. 落点或输出点：数据库查询结果、更新结果、文件系统、HTML 响应、异常信息和页面回显。

关键结论：

1. 阶段五不只是数据库阶段，而是“输入如何进入数据库、文件系统和输出点”的综合阶段。
2. 数据越权往往出现在持久层，不是因为用户没有登录，而是因为最终查询条件缺少资源归属约束。
3. 文件下载和上传场景的风险核心，在于用户是否能直接影响最终路径。
4. 输出点风险的核心，在于用户输入是否未经处理直接进入 HTML 或模板输出。
5. `MyBatis` 中 `#{}` 走参数绑定，`${}` 做字符串替换，审计时应优先搜索 `${}`。
6. `JPA/Hibernate` 虽然表现形式不同，但持久层审计主线仍然是“参数如何进入查询/更新条件、敏感字段如何绑定和暴露”。
7. 阶段五主体已经完成，后续只在 Spring 项目中继续看这些问题在现代框架中的落点。

重点风险点：

1. source：请求参数、表单字段、查询条件、排序字段、ID 参数、文件名、上传内容、输出文本。
2. sink：SQL 拼接、查询执行、更新执行、文件读取、文件写入、HTML 输出、Mapper XML 中的 `${}`。
3. 权限链或配置链：数据库连接信息、SQL 构造方式、资源归属约束是否下沉、路径是否规范化、输出前是否转义。

样例验证：

1. 验证目标：验证持久层越权、路径穿越、XSS 输出点和 `MyBatis` 参数绑定差异。
2. 样例路径或模块：`labs/lab-01-servlet-basics`
3. 现象记录：
   已加入订单查询脆弱/安全示例、文件上传下载脆弱/安全示例、XSS 脆弱/安全示例、`MyBatis #{}` / `${}` 对比例子，并完成本地编译和 Tomcat 部署同步。
4. 审计视角结论：
   当前已能把“用户输入 -> 持久层 / 文件系统 / 输出点”的多条风险链串起来分析。

1. 当前项目中的阶段五样例已足够做基础演示，但还没有扩展到更接近真实业务的 ORM 复杂场景。

### 阶段八：漏洞类型与 Source-Sink 映射

状态：进行中

学习目标：

1. 把常见漏洞拆成 `Source -> 中间处理链 -> Sink` 的固定分析结构。
2. 建立“有危险入口”与“可真正利用”之间的区别意识。
3. 在当前学习项目中逐步补齐最小漏洞演示代码，帮助从代码链路过渡到可观察现象。

本阶段已学习内容：

1. 已完成阶段八总入口学习，明确 `Source` 是外部可控输入、`Sink` 是危险落点、中间处理链是校验/转换/授权/持久层等中间环节。
2. 已完成 SQL 注入的第一轮 `Source-Sink` 映射，基于 `/api/users/vuln-list` 学习 `@RequestParam -> Service -> Mapper -> XML -> ${}` 的危险结构位进入链。
3. 已完成 XSS / 模板输出的第一轮 `Source-Sink` 映射，基于 `TemplateDemoController -> Model -> Thymeleaf` 理解输入进入 HTML 输出上下文的风险差异。
4. 已完成越权 / 资源归属问题的第一轮 `Source-Sink` 映射，明确“资源标识 + 当前身份”必须同时进入最终查询/更新条件。
5. 已在当前项目中补入 Java 原生反序列化最小学习链，新增 `/api/deser/**` 演示接口、`DemoSerializableNote`、`DemoHarmlessGadget` 和 `DeserializationDemoService`。
6. 已完成反序列化危险入口与安全对照的讲解，明确 `ObjectInputStream.readObject()` 是危险 Sink，`ObjectInputFilter` 白名单是核心收缩方式之一。
7. 已完成无害 gadget 教学示例，能通过 `DemoHarmlessGadget.readObject(...)` 演示“对象恢复时自动触发副作用”的基本机制。
8. 已完成反序列化审计思路第一轮讲解，明确 classpath、依赖树、版本和 gadget 候选条件之间的关系，并区分“有入口”与“有可用链”。

关键调用链或结构理解：

1. SQL 注入：`@RequestParam / @RequestBody -> Controller -> Service -> Mapper -> ${} / 原生 SQL 拼接`
2. XSS / 模板输出：`可控输入 -> Controller -> Model / 响应对象 -> HTML/模板输出位置`
3. 越权：`资源 ID + 当前用户 -> Controller -> Service -> Repository/Mapper -> 最终查询/更新条件`
4. 反序列化：`payload 字符串 -> Base64 decode -> ObjectInputStream.readObject() -> 类自己的 readObject(...) 副作用`

关键结论：

1. 阶段八和阶段七的区别在于：阶段七重点看框架和调用链，阶段八重点看漏洞如何落到具体 `Source`、中间处理链和 `Sink` 上。
2. SQL 注入的关键不是“输入进数据库”，而是“输入进入了 SQL 结构位置，且中间没有可靠约束”。
3. 越权问题的危险点常常不是某个危险函数，而是最终查询/更新条件没有把资源标识和当前身份同时收束。
4. 反序列化危险不在 JSON 绑定本身，而在业务代码把外部可控字节流交给对象恢复 API；是否可利用还要继续看 classpath 中的候选类、行为和版本匹配。
