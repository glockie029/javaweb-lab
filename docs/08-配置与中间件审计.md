# 配置与中间件审计

真实项目中的大量风险并不直接写在业务方法里，而是隐藏在配置文件、容器参数、中间件接入和运行环境之中。该文档用于补齐这部分能力。

## 1. 核心原则

1. 先找配置入口，再判断配置如何影响代码执行路径。
2. 先看“是否暴露”，再看“是否可利用”。
3. 不把配置审计和代码审计割裂开看。

## 2. 重点配置文件

优先检查以下文件：

1. `application.yml`
2. `application.properties`
3. `application-{profile}.yml`
4. `bootstrap.yml`
5. `logback.xml`
6. `web.xml`
7. 安全配置类和框架配置类
8. Docker、启动脚本、环境变量说明文件

## 3. 配置审计重点

### 3.1 敏感信息

1. 数据库账号口令
2. Redis、MQ、对象存储凭据
3. JWT 密钥、AES 密钥、第三方接口密钥
4. 默认管理员账号、测试账号

### 3.2 危险开关

1. 调试模式
2. 开发 profile
3. 关闭鉴权、关闭 CSRF、放宽 CORS
4. 开启调试端点、环境信息暴露、beans 暴露

### 3.3 外部资源加载

1. 模板路径
2. 文件存储路径
3. 插件目录
4. 外部配置目录
5. 脚本目录

审计关注点：

1. 是否可由攻击者控制路径或内容。
2. 是否加载自可写目录、网络路径或共享目录。

## 4. 容器与 Web 服务器

### 4.1 Tomcat

审计重点：

1. 部署方式和上下文路径
2. 默认管理接口和示例应用
3. 文件上传目录、临时目录、日志目录
4. 反向代理配合下的真实 IP 和转发头处理

### 4.2 反向代理

常见关注点：

1. 转发头信任
2. 内网接口是否被代理暴露
3. 静态目录、上传目录是否被直接映射

## 5. 数据库、缓存与消息中间件

### 5.1 数据库

1. 是否暴露测试库或开发库连接。
2. 是否在错误日志中打印连接信息。
3. 是否存在高权限账号直连业务库。

### 5.2 Redis

1. Session、验证码、缓存对象是否存储在 Redis。
2. 是否存在把对象直接序列化进缓存的设计。
3. 是否存在配置暴露、未授权访问或跨环境复用。

### 5.3 消息队列

1. 消息消费者是否做输入校验。
2. 消息体是否被反序列化为危险对象。
3. 是否存在内部消息接口被外部间接触发。

## 6. 文件系统与对象存储

1. 上传目录、导出目录、模板目录是否隔离。
2. 对象存储回调、签名、公开读策略是否安全。
3. 是否把导出文件、临时文件存到 Web 可访问目录。

## 7. 日志与监控

1. 日志是否记录口令、Token、验证码、PII。
2. 错误页面和异常处理是否暴露堆栈、SQL、路径、版本信息。
3. 监控系统、日志检索系统、管理端点是否开放在外网。

## 8. 环境与运行参数

1. 启动参数是否包含敏感配置。
2. JVM 参数是否暴露调试、代理或本地类路径信息。
3. 部署脚本是否包含明文密钥、固定路径和调试命令。

## 9. 实战审计顺序

建议按以下顺序执行：

1. 先枚举所有配置文件与 profile。
2. 再枚举外部服务连接与凭据。
3. 再检查调试端点、管理端点、上传目录和静态目录。
4. 最后把配置结果与代码中的入口、权限、文件处理逻辑对应起来。
