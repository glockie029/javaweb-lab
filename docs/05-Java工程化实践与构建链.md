# Java 工程化实践与构建链

该文档用于补齐 Java 项目工程化基础。对代码审计来说，构建链和依赖结构是进入真实项目的第一道门。

## 1. 为什么工程化必须纳入审计学习

1. 依赖和版本决定了项目的组件边界与已知风险面。
2. 多模块结构决定了真正的入口模块、公共安全模块和共享工具模块在哪里。
3. 打包方式和配置加载方式决定了实际暴露面、部署形态和复现条件。
4. 插件、profile 和资源过滤可能改变最终运行时行为。

## 2. Maven 基础

### 2.1 需要掌握的概念

1. 坐标：`groupId`、`artifactId`、`version`
2. 生命周期：`clean`、`compile`、`test`、`package`、`install`
3. 依赖范围：`compile`、`provided`、`runtime`、`test`
4. 传递依赖、依赖冲突、版本仲裁
5. 聚合与继承

### 2.2 必看位置

1. `pom.xml` 顶层结构
2. `dependencies`
3. `dependencyManagement`
4. `plugins`
5. `profiles`
6. `properties`
7. `modules`

### 2.3 审计关注点

1. 是否使用高风险组件或过旧版本。
2. 是否存在测试依赖、调试依赖、管理依赖被打入产物。
3. 是否有 profile 导致不同环境行为明显不同。
4. 是否存在插件修改打包结果、资源内容或类路径。

## 3. Gradle 基础

### 3.1 需要掌握的概念

1. `plugins`
2. `dependencies`
3. `tasks`
4. 多模块项目中的 `settings.gradle`
5. 子项目与根项目的配置继承

### 3.2 审计关注点

1. 构建脚本是否动态引入依赖或配置。
2. 是否存在本地脚本、私有仓库、自定义任务改变运行行为。
3. 是否把测试、调试或开发资源打进最终构建产物。

## 4. 多模块工程阅读方法

真实项目通常不是单模块。建议按以下顺序识别：

1. 找父工程或根工程。
2. 找启动模块。
3. 找公共模块，如 `common`、`core`、`security`、`web`、`api`、`dao`。
4. 找管理端、开放接口端、任务模块、网关模块。
5. 画出模块依赖关系，标出安全逻辑集中点。

审计关注点：

1. 公共鉴权是否在共享模块中统一实现。
2. 业务模块是否绕过共享安全逻辑。
3. 各模块是否引用不同版本的风险组件。

## 5. 打包方式与运行形态

### 5.1 常见形态

1. `war` 部署到外部 `Tomcat`
2. `jar` 方式运行 `Spring Boot`
3. 多服务拆分
4. 管理后台、开放接口、定时任务分模块部署

### 5.2 审计关注点

1. 容器外置还是内嵌，配置位置是否不同。
2. 上传目录、静态资源目录、日志目录是否落在可访问路径。
3. 打包后是否暴露测试资源、示例配置、脚本文件。

## 6. 配置加载与环境隔离

需要重点理解：

1. `application.yml`、`application-{profile}.yml`
2. 环境变量和启动参数
3. 外部配置文件覆盖
4. 敏感配置注入

审计关注点：

1. 是否存在默认口令、硬编码密钥、测试数据库连接。
2. 是否存在 `dev`、`test` profile 暴露额外接口或关闭安全校验。
3. 是否允许从外部路径加载配置、模板、脚本或资源。

## 7. 插件与构建产物审计

重点关注插件：

1. `spring-boot-maven-plugin`
2. `maven-war-plugin`
3. `maven-shade-plugin`
4. 代码生成、资源复制、模板预处理类插件

审计关注点：

1. 是否改变类路径、打包边界、资源路径。
2. 是否把内部资源、测试资源、脚本资源打入产物。
3. 是否把配置值在构建期直接替换进资源文件。

## 8. 实战阅读顺序

进入项目后建议先做以下事情：

1. 看根目录是否有 `pom.xml`、`build.gradle`、`settings.gradle`。
2. 找启动模块、父模块和安全相关模块。
3. 枚举所有核心依赖和版本。
4. 判断项目是传统 `war` 还是现代 `jar`。
5. 找 profile、配置文件、插件和外部资源引用。

## 9. 阶段输出建议

1. 形成一份“常见依赖 -> 技术栈 -> 风险方向”的对照表，至少覆盖 `Spring`、`Shiro`、`Fastjson`、`Jackson`、模板引擎、日志组件。
2. 对每个实际项目先画工程结构图，再开始审计业务代码。
3. 把高频风险组件单独列入项目检查表。
