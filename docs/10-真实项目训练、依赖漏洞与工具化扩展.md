该文档用于补充现有 JavaWeb 代码审计学习路线中较容易在主线文档里显得臃肿的内容，重点补齐：

1. 从学习样例过渡到真实项目的训练分级。
2. 第三方依赖漏洞识别与可达性分析。
3. 历史漏洞补丁 diff 复盘方法。
4. SAST / 半自动审计工具的使用边界与记录方式。

建议使用方式：

1. 先按 [01-JavaWeb前置学习路线.md](/Users/mac/Documents/javaweb-lab/docs/01-JavaWeb前置学习路线.md) 完成主阶段学习。
2. 进入真实项目时，再按本文档的 `L1 -> L5` 顺序推进训练强度。
3. 每做一个真实项目，同时补做依赖识别、补丁 diff 或工具化扫描中的至少一项。

## 1. 真实项目训练分级

### 1.1 模块定位

用于补齐从“学习样例”过渡到“真实项目审计”的训练路径，避免一开始直接进入过大项目，导致无法建立稳定的审计方法。

### 1.2 L1：自写 Servlet + JDBC 小项目

补充目标：

1. 训练传统 JavaWeb 基础请求链。
2. 训练 Servlet、Filter、Session、JDBC、文件处理的基础审计能力。
3. 建立最小 source-sink 分析习惯。

补充重点：

1. 登录与 Session。
2. JDBC 查询与 SQL 拼接。
3. 文件上传与下载。
4. 简单后台接口。
5. 异常信息输出。
6. 水平越权示例。

建议输出物：

1. Servlet 请求链图。
2. JDBC SQL 执行链图。
3. SQL 注入审计记录。
4. 水平越权审计记录。
5. 文件下载路径穿越审计记录。

### 1.3 L2：自写 Spring Boot + MyBatis 项目

补充目标：

1. 训练现代 JavaWeb 主流调用链。
2. 理解 Controller、Service、Mapper、MyBatis XML 的关系。
3. 训练 Spring 参数绑定、拦截器、权限校验和动态 SQL 审计。

补充重点：

1. `@RestController`
2. `@RequestMapping`
3. `@RequestParam`
4. `@PathVariable`
5. `@RequestBody`
6. DTO 参数绑定。
7. Interceptor / Filter。
8. MyBatis `${}` 与 `#{}`。
9. 动态排序、动态表名、动态条件。
10. 登录校验与资源归属校验。

建议输出物：

1. Spring MVC 请求链图。
2. MyBatis 风险映射表。
3. 登录鉴权链路笔记。
4. 动态 SQL 审计记录。
5. 水平越权案例复盘。

### 1.4 L3：小型开源 CMS / 后台系统

补充目标：

1. 从自写样例进入真实开源项目。
2. 训练真实工程结构识别、配置识别、入口枚举和漏洞链追踪。
3. 建立完整项目审计复盘能力。

项目选择标准：

1. Java / Spring Boot 技术栈。
2. 单体项目或简单多模块。
3. 有后台管理功能。
4. 有用户、角色、权限模块。
5. 有文件上传或内容管理功能。
6. 有数据库脚本。
7. 能本地运行。
8. 代码量适中。
9. 最好有历史漏洞、issue 或补丁记录。

建议输出物：

1. 技术栈识别报告。
2. 路由入口清单。
3. 权限链说明。
4. 高风险接口清单。
5. 3 至 5 个完整问题记录。
6. 项目审计复盘报告。

### 1.5 L4：中型多模块 Spring Boot 项目

补充目标：

1. 训练多模块项目阅读能力。
2. 训练公共安全模块、权限模块、工具模块和业务模块之间的关系分析。
3. 训练中型项目中的权限链、配置链、依赖链和公共组件审计。

补充重点：

1. 父工程。
2. 启动模块。
3. 公共模块。
4. 安全模块。
5. 后台管理模块。
6. 开放 API 模块。
7. 定时任务模块。
8. 文件服务模块。
9. 数据访问模块。
10. 代码生成模块。

建议输出物：

1. 模块关系图。
2. 安全模块分析笔记。
3. 公共组件风险清单。
4. 权限链审计报告。
5. 数据权限问题记录。
6. 多模块项目复盘报告。

### 1.6 L5：历史 CVE / CNVD 项目复盘

补充目标：

1. 通过历史漏洞学习真实漏洞成因。
2. 训练补丁 diff 分析。
3. 训练同类漏洞变体搜索。
4. 提升 CVE / CNVD 挖掘能力。

项目选择标准：

1. 有公开漏洞编号。
2. 有明确影响版本。
3. 有漏洞版本和修复版本。
4. 有公开源码仓库。
5. 有 commit diff 或 release diff。
6. 有较清晰触发路径。
7. 技术栈与 JavaWeb / Spring 生态相关。

建议输出物：

1. 漏洞成因分析。
2. 补丁 diff 分析。
3. source-sink 调用链。
4. 复现条件。
5. 修复方式。
6. 同类变体搜索记录。
7. 完整漏洞复盘报告。

## 2. 依赖漏洞识别

### 2.1 模块定位

用于补齐 Java 项目中第三方组件、传递依赖、历史漏洞和漏洞可达性分析能力。

### 2.2 依赖识别入口

Maven 重点位置：

1. `pom.xml`
2. 父级 `pom.xml`
3. 子模块 `pom.xml`
4. `dependencyManagement`
5. `dependencies`
6. `plugins`
7. `profiles`
8. `properties`

Gradle 重点位置：

1. `build.gradle`
2. `build.gradle.kts`
3. `settings.gradle`
4. `gradle.properties`
5. 根项目配置。
6. 子项目配置。

### 2.3 高频风险组件方向

需要重点覆盖以下组件类型：

1. JSON 组件：Fastjson、Jackson、Gson。
2. 安全框架：Shiro、Spring Security。
3. 模板引擎：FreeMarker、Velocity、Thymeleaf、JSP / EL。
4. Web 框架：Spring MVC、Struts2。
5. ORM：MyBatis、Hibernate、JPA。
6. XML 解析组件：dom4j、SAX、JAXP、XStream。
7. 日志组件：Log4j、Logback。
8. 文件处理组件：Commons FileUpload、Zip4j、POI。
9. HTTP 客户端：HttpClient、OkHttp、RestTemplate、WebClient。
10. 缓存与序列化组件：Redis、Kryo、Hessian、Java serialization。

### 2.4 依赖漏洞分析流程

建议固定按以下顺序判断：

1. 确认组件是否存在。
2. 确认版本是否受影响。
3. 确认是直接依赖还是传递依赖。
4. 确认是否被打入最终产物。
5. 确认危险功能是否启用。
6. 确认项目中是否调用危险 API。
7. 确认外部输入是否能进入危险路径。
8. 确认是否需要特殊配置、特殊权限或特殊环境。
9. 确认可达性。
10. 输出风险结论。

### 2.5 依赖漏洞结论分级

建议使用以下结论类型：

1. 确认可利用。
2. 存在风险但未证实可利用。
3. 仅存在受影响依赖。
4. 不受影响。

### 2.6 依赖漏洞记录模板

建议记录以下字段：

1. 组件名称。
2. groupId。
3. artifactId。
4. 当前版本。
5. 引入方式。
6. 所属模块。
7. 漏洞编号。
8. 漏洞类型。
9. 影响版本。
10. 修复版本。
11. 是否命中影响版本。
12. 是否被打入最终产物。
13. 外部输入点。
14. 关键调用链。
15. 危险 API。
16. 权限要求。
17. 配置前提。
18. 利用条件。
19. 风险结论。
20. 修复建议。
21. 回归验证点。

## 3. 补丁 diff 复盘

### 3.1 模块定位

用于通过真实漏洞修复记录学习漏洞成因、修复方式、同类变体挖掘和复盘输出。

### 3.2 适合复盘的漏洞类型

建议优先覆盖：

1. SQL 注入。
2. 权限绕过。
3. 未授权访问。
4. 文件上传绕过。
5. 任意文件读取。
6. 路径穿越。
7. SSRF。
8. XXE。
9. 反序列化。
10. 表达式注入。
11. 模板注入。
12. 认证绕过。
13. 敏感信息泄露。

### 3.3 补丁 diff 复盘流程

建议按以下步骤执行：

1. 确定漏洞版本和修复版本。
2. 获取漏洞版本源码。
3. 获取修复版本源码。
4. 对比 commit diff 或 release diff。
5. 定位修改文件。
6. 分析修改前代码。
7. 分析修改后代码。
8. 还原漏洞调用链。
9. 判断 source、validation、sink。
10. 判断漏洞触发条件。
11. 判断修复方式是否完整。
12. 总结同类搜索关键词。
13. 搜索同类变体。
14. 输出复盘报告。

### 3.4 修改文件重点关注位置

需要重点关注：

1. Controller。
2. Service。
3. DAO / Mapper。
4. Filter。
5. Interceptor。
6. Security Config。
7. Template。
8. File Utility。
9. JSON / XML Parser。
10. Auth / Permission Logic。
11. Dependency Version。
12. Config 文件。

### 3.5 修改前代码分析维度

建议固定回答以下问题：

1. source 从哪里进入。
2. source 是否外部可控。
3. 参数如何传递。
4. 是否存在校验。
5. 是否存在权限判断。
6. sink 是什么。
7. 为什么原逻辑不安全。
8. 哪些条件下可以触发。
9. 影响范围是什么。

### 3.6 修改后代码分析维度

建议固定回答以下问题：

1. 是否增加白名单。
2. 是否增加权限校验。
3. 是否增加资源归属校验。
4. 是否增加路径规范化。
5. 是否禁用危险协议。
6. 是否禁用外部实体。
7. 是否改为预编译。
8. 是否移除危险 API。
9. 是否升级依赖版本。
10. 是否增加签名校验。
11. 是否修改默认配置。
12. 是否增加测试用例。

### 3.7 补丁 diff 复盘模板字段

建议记录以下字段：

1. 项目名称。
2. 项目地址。
3. 漏洞编号。
4. 漏洞类型。
5. 漏洞版本。
6. 修复版本。
7. 修复 commit。
8. 影响模块。
9. 影响接口。
10. 修改文件。
11. 修改类型。
12. 修改前 source。
13. 修改前 sink。
14. 漏洞调用链。
15. 修复方式。
16. 修复边界。
17. 复现条件。
18. 同类搜索关键词。
19. 同类变体搜索结果。
20. 修复建议。
21. 回归验证点。
22. 复盘结论。

## 4. SAST / 半自动审计工具使用

### 4.1 模块定位

用于补齐代码审计中的工具化、半自动化和批量排查能力，但工具结果必须经过人工复核，不能直接等同于漏洞结论。

### 4.2 工具适合做的事情

适合覆盖：

1. 快速枚举危险 API。
2. 发现常见 source-sink 模式。
3. 扫描依赖漏洞。
4. 发现硬编码敏感信息。
5. 辅助定位调用链。
6. 批量检查同类代码。
7. 复核补丁是否覆盖同类问题。

### 4.3 工具不适合直接判断的事情

不应直接依赖工具结论判断：

1. 复杂业务越权。
2. 资源归属关系。
3. 真实权限边界。
4. 业务状态机漏洞。
5. 漏洞是否真实可利用。
6. 实际影响范围。
7. 修复是否符合业务安全边界。

### 4.4 推荐工具分类

建议重点覆盖：

1. `Semgrep`：规则扫描、危险 API、常见模式。
2. `CodeQL`：语义查询、数据流分析、变体搜索。
3. `SpotBugs / FindSecBugs`：Java 静态安全扫描。
4. `OWASP Dependency-Check`：依赖 CVE 扫描。
5. Maven dependency tree / Gradle dependencies：依赖树分析。
6. `Gitleaks / TruffleHog`：密钥扫描。
7. `IntelliJ IDEA`：调用层级、全局搜索、类型跳转。
8. `git diff / git blame / git log`：补丁、历史变更和风险代码来源分析。

### 4.5 Semgrep 使用方向

建议补充以下场景：

1. 扫描危险 API。
2. 编写项目专项规则。
3. 扫描命令执行点。
4. 扫描反序列化点。
5. 扫描文件操作点。
6. 扫描 SSRF 外连点。
7. 扫描 MyBatis `${}`。
8. 扫描不安全 XML 解析。
9. 扫描硬编码敏感信息。
10. 对结果进行人工确认。

### 4.6 CodeQL 使用方向

建议补充以下场景：

1. 创建 Java 项目数据库。
2. 运行官方 Java 查询包。
3. 编写自定义 query。
4. 追踪 source 到 sink。
5. 做变体漏洞搜索。
6. 分析 SQL 注入、路径穿越、命令执行、SSRF、XXE、反序列化等风险。
7. 对结果结合权限和业务场景人工复核。

### 4.7 SpotBugs / FindSecBugs 使用方向

建议补充以下场景：

1. Java 常见安全缺陷扫描。
2. SQL 注入风险识别。
3. 路径穿越风险识别。
4. XXE 风险识别。
5. 不安全随机数识别。
6. 硬编码密码识别。
7. 不安全 Cookie 识别。
8. 不安全加密算法识别。
9. 结果误报复核。

### 4.8 OWASP Dependency-Check 使用方向

建议补充以下场景：

1. 扫描第三方依赖 CVE。
2. 识别组件版本。
3. 识别 CVSS 分数。
4. 识别影响说明。
5. 识别修复版本。
6. 结合依赖调用链判断可达性。
7. 区分“存在漏洞依赖”和“确认可利用”。
